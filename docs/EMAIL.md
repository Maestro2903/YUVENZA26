# Confirmation emails on Render - setup guide

The confirmation-email sender is a small Node service
(`google-cloud/email-function/`) that the Next.js app calls after every
successful payment. It renders the branded template (stamp logo, entries
with dates and times, amount), generates the signed QR pass as an inline
PNG, and sends the mail through **Gmail + Nodemailer**. It is protected by
a shared secret header, and sending is idempotent - an atomic claim on
`orders.confirmation_email_sent_at` means retries never double-send.

This guide hosts it on **Render** (free tier, no CLI). The Google Cloud
Functions alternative lives in
[google-cloud/email-function/README.md](../google-cloud/email-function/README.md).

## How it fits together

```
Payment verified (site / webhook)
        │
        ▼
lib/email/notify.ts  ── claims orders.confirmation_email_sent_at
        │  POST { order details, QR payload }  +  x-email-secret header
        ▼
Render web service "yuvenza-email"  (google-cloud/email-function/)
        │  Nodemailer + Gmail app password
        ▼
Attendee's inbox: branded email, scannable QR, PDF-download button
```

- Unset email env vars = sending is silently skipped; payments are never
  affected by email failures.
- If a send fails, the claim is released so the Razorpay webhook retry
  sends it again.

## 1. Gmail app password (one time)

1. Use (or create) the Gmail account the mails should come from,
   e.g. `yuvenza.cit@gmail.com`.
2. Turn on **2-Step Verification**: myaccount.google.com → Security.
3. Create an **App password**: myaccount.google.com/apppasswords →
   name it `yuvenza-email` → copy the 16-character password.

> Gmail sends ~500 recipients/day on a normal account - fine for a fest.
> If you outgrow it, swap the transport for Brevo/Resend later; the
> template and the app contract stay identical.

## 2. Create the Render service (one time, ~5 minutes)

The repo root has a `render.yaml` blueprint that configures everything.

1. Sign in at **render.com** with the GitHub account that owns this repo
   (`Maestro2903` - the repo is private under it).
2. **New → Blueprint** → select `Maestro2903/YUVENZA26` → **Apply**.
   Render creates a free web service named **`yuvenza-email`** from
   `google-cloud/email-function/`.
3. Open the service → **Environment** tab:
   | Key | Value |
   |---|---|
   | `GMAIL_USER` | the Gmail address from §1 |
   | `GMAIL_APP_PASSWORD` | the 16-character app password |
   | `EMAIL_SHARED_SECRET` | already auto-generated by Render - **copy it** for §3 |
   | `FROM_NAME` | already set (`Yuvenza · The Youth Club`) |
4. Wait for the deploy to go **Live**, then note the service URL,
   e.g. `https://yuvenza-email.onrender.com`.

## 3. Point the site at it

In the **Vercel** project (Settings → Environment Variables → Production):

| Key | Value |
|---|---|
| `EMAIL_FUNCTION_URL` | the Render service URL from §2.4 |
| `EMAIL_FUNCTION_SECRET` | the `EMAIL_SHARED_SECRET` value copied in §2.3 |

Then **redeploy** the site (env vars only apply to new deployments).

## 4. Test it

1. Health check: open the service URL in a browser →
   `{"ok":true,"service":"yuvenza-email"}`.
2. End to end: make a demo registration on the site → the confirmation
   email should arrive within a minute (first one after idle can be slow,
   see below). Or use **Admin → Payments → Resend email** on a paid order.
3. Preview the template offline anytime:
   `node scripts/email-demo.mjs` → HTML + PDF land in `~/Downloads`.

## Free-tier caveat and keep-warm

Render's free tier **sleeps the service after ~15 idle minutes**; the
first request after that takes up to a minute to answer. Registrations
are never blocked by this (email is fire-and-forget with retry), but
during fest days keep it warm: point a free uptime monitor (e.g.
UptimeRobot) at `GET /` every 10 minutes. The health endpoint needs no
secret and sends nothing.

## Troubleshooting

| Symptom | Check |
|---|---|
| No emails at all | `EMAIL_FUNCTION_URL`/`EMAIL_FUNCTION_SECRET` set on Vercel? Redeployed after setting them? |
| `401 unauthorized` in Render logs | The two secrets don't match - re-copy `EMAIL_SHARED_SECRET` from Render into Vercel |
| `535` / auth errors in Render logs | Wrong app password, or 2-Step Verification is off on the Gmail account |
| First email very slow | Free-tier cold start - add the uptime monitor above |
| One order never got its email | Admin → Payments → that order → **Resend email** |

## Changing the sender later

The service is plain Nodemailer - to move off Gmail, change the
transport in `google-cloud/email-function/index.js` (e.g. Brevo or
Resend SMTP), update the env vars on Render, and nothing on the site
changes.
